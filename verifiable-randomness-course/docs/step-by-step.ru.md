## Зачем этот курс

Большинство лотерей и «случайных» розыгрышей доверяют оператору, который может тихо перекрутить невыгодный результат. Курс показывает криптографию, которая убирает это доверие: верифицируемый chaos-розыгрыш, функцию проверяемой задержки (VDF), доказывающую затраченную работу, доказательство ECVRF, проверяемое офлайн всего из **80 байт**, и time-lock RSW, скрывающий значение до тех пор, пока его не раскроет доказуемая работа. Затем вы **соберёте смещённую лотерею, сломаете её и почините** розыгрышем, который оператор не может перебирать — ровно тот паттерн, что потребляют контракты AIMarket `lottery/` на Base.

Каждая лаба обращается к **реальному оракулу AIMarket** — Platon, Chronos, Sortes, Aestus — запущенному локально, а не к моку. Каждая печатает блок `trace`, чтобы верифицируемый артефакт (доказательство, beta, puzzle, вектор VDF) можно было изучить, а не принимать на веру.

## Установка (10 минут)

### Шаг 1 — Клон и установка

Точные команды быстрого старта (обратите внимание на extra `[oracles,dev]` — `oracles` тянет Platon/Chronos/Sortes/Aestus, `dev` — pytest):

```bash
git clone https://github.com/alexar76/aimarket-courses.git
        cd aimarket-courses/verifiable-randomness-course
pip install -e ".[oracles,dev]"
```

**Из монорепы aicom** (рекомендуется для контрибьюторов): пакеты оракулов подключаются автоматически по соседним путям в `oracles/` — `courselib/oracle_paths.py` поднимается до корня монорепы и добавляет `oracles/oracles/{chronos,sortes,aestus}` и `oracles/oracles/platon/backend` в `sys.path`. Кроме `numpy` для Platon, ничего ставить не нужно.

```bash
cd courses/verifiable-randomness-course
pip install -e ".[oracles,dev]"
```

### Шаг 2 — Тесты

```bash
pytest -q
```

**Ожидается:** все тесты зелёные (паритет i18n en/ru/es, хелперы случайности, проверки упражнений).

Строки UI (заголовки, метки `доказательство`/`проверка`/`трейс`, названия билетов) берутся из `i18n/{en,ru,es}.json`. Переключение через `COURSE_LANG`:

## M1 — Chaos VRF с Platon

**Концепция:** непредсказуемая случайность от верифицируемого chaos-оракула — chaos-state + client seed + энтропия ОС, связанные подписью Ed25519.
**Лаба:** `labs/lab01_platon_chaos.py` · **~15 мин** · нужен `numpy` + Platon

### Шаги

1. Прочитайте docstring в начале `labs/lab01_platon_chaos.py` (отрасль: лотереи, гейминг).
2. `python labs/lab01_platon_chaos.py`
3. В выводе: схема `доказательство`, первые байты розыгрыша, результат `проверка`.
4. `courselib/randomness.py` → `chaos_draw()`: временный `Signer`, chaos-`state`, `draw_randomness(...)`, затем `verify_randomness(...)`.

### Ожидаемый вывод

- `proof: scheme=platon-chaos-vrf/v1`, подписанный розыгрыш **32 байта**, `verify: True`
- блок `trace` с событием `chaos_draw` (`verified=True, bytes=32`)

### Самопроверка

- [ ] Объясняете, почему client seed + подпись делают розыгрыш непредсказуемым
- [ ] Видите 32 байта и `verify: True` против публичного ключа

## M2 — VDF-время с Chronos

**Концепция:** Wesolowski VDF доказывает затраченную *последовательную* работу в группе неизвестного порядка RSA-2048 — доказательство затраченной работы, а не доверие к системным часам.
**Лаба:** `labs/lab02_chronos_vdf.py` · **~20 мин** · нужен M1

### Шаги

1. `python labs/lab02_chronos_vdf.py`
2. Строка `доказательство`: proof `pi` и простое `l`; `verify: True`.
3. **Демо подделки:** лаба вызывает `vdf.verify(g, y+1, …)` на испорченном выходе — печатает `False`.
4. `courselib/randomness.py` → `vdf_eval()`: `hash_to_group` → `evaluate` → `prove` → `verify`. Используется `difficulty=700` (~700 возведений в квадрат) — считается за секунды.

### Ожидаемый вывод

- `proof: pi=…… l=<простое>`, `verify: True`, `проверка подделанного VDF: False`
- `trace` с событиями `vdf_eval` и `tamper_check`

### Самопроверка

- [ ] Формулируете, что VDF доказывает (последовательную работу) и что **не** доказывает (реальное время)
- [ ] Видели отказ проверки на подделанном выходе

## M3 — ECVRF и Sortes

**Концепция:** доказательство ECVRF по RFC 9381. Для фиксированной пары (публичный ключ, alpha) существует ровно **одна** валидная beta — оператор не может перебирать выгодный выход. Доказательство — **80 байт** и проверяется офлайн, без повторного обращения к оракулу.
**Лаба:** `labs/lab03_sortes_ecvrf.py` · **~20 мин** · нужен M1

### Шаги

1. `python labs/lab03_sortes_ecvrf.py`
2. Убедитесь, что доказательство ровно **80 байт** (`80 bytes = <hex>…`).
3. Отметьте `beta` (верифицируемый случайный выход) и `verify: True`.
4. `courselib/randomness.py` → `ecvrf_draw()`: `sk_to_pk` → `prove(sk, alpha)` → `verify(pk, alpha, pi)`.

### Ожидаемый вывод

- `proof: 80 bytes = <hex>…`, `result: beta=<hex>…`, `verify: True`
- `trace` с событием `ecvrf` (`proof_len=80, verified=True`)

### Самопроверка

- [ ] Объясняете «одна beta на (pk, alpha)» и почему это блокирует перебор
- [ ] Проверили, что доказательство **80 байт** (число офлайн-проверки из тэглайна)

## M4 — Time-lock RSW с Aestus

**Концепция:** головоломка time-lock RSW. Запечатайте значение сейчас; открыть его можно только после `T` последовательных возведений в квадрат — зашифруйте сейчас, расшифруйте после доказуемой задержки. Кирпич для запечатанных ставок и commit-reveal, где никто не должен подглядывать раньше времени.
**Лаба:** `labs/lab04_aestus_timelock.py` · **~20 мин** · нужен M2

### Шаги

1. `python labs/lab04_aestus_timelock.py`
2. Цикл seal → open: `T=350` возведений, затем открытый payload и `verify: True`.
3. `courselib/randomness.py` → `timelock_seal()`: `rsw.seal(message, T=...)` → `rsw.open_puzzle(puzzle)`; `message_match` подтверждает целостность открытого текста.

### Ожидаемый вывод

- `proof: T=350 squarings`, `result: <открытый payload>`, `verify: True`
- `trace` с событием `timelock` (`T=350, opened=True`)

### Самопроверка

- [ ] Объясняете, почему time-lock скрывает значение без доверенной третьей стороны
- [ ] Видели восстановление запечатанного сообщения после `T` возведений

## M5 — Commit–reveal и атаки смещения

**Концепция:** ключевой модуль. Соберите лотерею, которую нечестный оператор может перебирать, покажите смещение, затем замените розыгрыш на Sortes ECVRF, который оператор перебрать **не может**.
**Лаба:** `labs/lab05_commit_reveal.py` · **~25 мин** · трек: экономика · нужен M3

### Шаги

1. `python labs/lab05_commit_reveal.py`
2. Сравните билеты: `смещённый билет` vs `честный VRF-билет`. Строка `verify VRF: True` доказывает, что честный розыгрыш несёт верифицируемое доказательство.
3. Прочитайте **аудит-заметку**: «Честный розыгрыш привязывает одну beta к каждому seed — оператор не может перебирать.»
4. `courselib/randomness.py` → `BiasedLottery` (до 200 попыток перебора `operator_secret|player_seed|attempt` ради джекпота) vs `FairLottery` (один розыгрыш ECVRF → один билет, перебор невозможен).

### Ожидаемый вывод

- `смещённый билет: <n>` и `честный VRF-билет: <n>`, `verify VRF: True`, строка аудит-заметки
- `trace` с событиями `commit` и `draw` (`vrf_ok=True`)

### Самопроверка

- [ ] Описываете атаку перебором, которую делает `BiasedLottery`
- [ ] Объясняете, почему `FairLottery` (одна beta на seed) к ней невосприимчива

**Лаба:** `labs/lab06_onchain_consumer.py` · **~20 мин** · трек: продвинутый · нужны M2, M3

## Чтение трейса (в каждой лабе)

**Концепция:** каждая лаба пишет небольшой `Trace` (см. `courselib/trace.py`), чтобы поведение агента/оракула можно было изучить.

### Шаги (после каждой лабы)

1. Найдите блок `trace (<n> событий):` внизу вывода.
2. Прочитайте каждое событие — `kind` плюс залогированные поля (`chaos_draw` → `verified, bytes`; `vdf_eval` → `difficulty, valid`; `tamper_check` → `valid=False`).
3. Сопоставьте каждое событие со строкой stdout выше. Трейс — это «лог юнит-теста» верифицируемой случайности.

### Самопроверка

- [ ] Показываете событие, доказывающее, что розыгрыш прошёл проверку
- [ ] В M2 находите событие `tamper_check` со `valid=False`

## Упражнения и сертификат

После лаб запустите DIY-проверки. У каждого модуля своё упражнение в `courselib/exercises.py` (M2 подделывает VDF и проверяет, что verify падает; M3 проверяет ровно 80 байт; M5 — что честный розыгрыш проходит проверку):

```bash
python labs/run_exercises.py                       # все шесть, ✓/✗ по модулям
python labs/run_exercises.py --module m3           # проверка одного модуля
```

Когда все шесть зелёные — выпустите сертификат:


Создаётся `certificate.html` — откройте в браузере и **Print → Save as PDF**. ID удостоверения детерминированно выводится из имени + даты выпуска (`courselib/certificate.py`). Сертификат блокируется до прохождения упражнений, если не передать `--skip-check`.

## Типичные проблемы

| Проблема | Решение |
| --- | --- |
| `No module named 'chronos'` | `pip install -e ".[oracles]"` или запуск из монорепы aicom |
| `No module named 'oracle_core'` | `oracles/core` должен быть на пути — авто в монорепе; тянется `[oracles]` для Sortes/Platon |
| `Oracle package '…' not installed` | вне монорепы и без extra — поставьте `.[oracles]` или клонируйте рядом с `oracles/` |
| Сбой импорта Platon | поставьте `numpy`; проверьте `oracles/oracles/platon/backend` |
| Медленная VDF-лаба | лабы используют ~700 возведений (M2) / `T=350` (M4) — снижайте сложность только в своём упражнении |
| Пустые строки RU/ES | `export COURSE_LANG=ru\|es`; ключи `i18n/ru.json` должны совпадать с `i18n/en.json` |
| Ошибка i18n в `pytest` | добавьте недостающий ключ во **все три** каталога (en/ru/es) |
| Старый код в Colab | перезапустите setup-ячейку; зафиксируйте ветку `main` |
